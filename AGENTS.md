# AGENTS.md

This file provides guidance to coding agents (e.g. Claude Code) when working with code in this repository. (`CLAUDE.md` is a symlink to this file.)

## What rookery is

An orchestrator agent with memory. A resident **daemon** hosts **master agent** sessions built on the Claude Agent SDK, and the master spawns, controls, and observes a **worker fleet isolated per worktree** of registered git repos. The CLI, the Electron app, and Slack are all **thin clients** that attach to the daemon over WebSocket — the daemon survives client disconnects and does not kill sessions or workers.

## Glossary

- **Master (`MasterAgent`)** — the orchestrator you talk to. One session = one master conversation. Events `master.*`.
- **Worker (`Worker`)** — a worker the master spawns in isolation inside a worktree. Managed by `FleetOrchestrator`, DB `workers`, events `worker.*`. (The current name for the old term "subagent.")
- **Fleet** — the entire set of workers = the control plane (one global instance per daemon).
- **Session / Repo / Worktree** — the master conversation / a registered git repo / a per-worker isolated workspace.
- **Native nested subagent** — a subagent the worker's own provider spawns natively (live-only, non-persistent): on Claude, an SDK subagent launched via the **Task tool** of `claude_code`; on Codex, a **collab child thread** (`multi_agent` feature, mapped in `codex-backend.ts` — see docs/superpowers/specs/2026-07-11-codex-nested-agents-design.md). Both flow as `worker.nested` events + desktop `NestedAgents`, panel-keyed by `parentToolUseId` (= Task call id on Claude / child threadId on Codex). ⚠️ Codex collab children can outlive the parent turn and their token spend is NOT billed into `cumCostUsd` (the `costBudgetUsd` guard can't see them — recorded limitation). ⚠️ **The words "agent"/"subagent" are reserved for this native concept** — rookery workers are always called "worker/Worker" (`forwardSubagentText` and `subagent_type` are SDK tokens, so they are preserved).

## Required: Node 22 (the most common pitfall)

`better-sqlite3` is built against the **Node 22 ABI (127)**. This machine defaults to an older Node, so **whatever you do, activate Node 22 first** (`nvm use 22` or PATH). If you get it wrong the daemon "won't come up" and an ABI mismatch is logged to `~/.rookery/daemon.log`. To work around this, the Electron app spawns the daemon as an **external Node 22 process** (`ROOKERY_NODE`) — which is why the app's main/renderer **must not runtime-import** `better-sqlite3` or daemon code.

## Architecture (the big picture you only see by reading several files)

This section is the always-loaded summary. For depth — turn lifecycles, the event/protocol/schema catalogs, and "how to add X" recipes — read the on-demand docs under [`docs/`](docs/README.md) (each cites its source files).

### Transport-agnostic core + a single composition root
`src/core/` **never imports** WS/CLI/Slack/Electron. It is driven by domain commands and only emits `CoreEvent` through the `EventBus`. Every external boundary is an **injectable port + fake**: **AgentBackend** (provider-neutral agent port; `ClaudeBackend` adapts the SDK `query()` — the raw `QueryFn` remains only for Claude-specific aux paths: labeler, command probe), `GitOps` (`RealGitOps`/`FakeGitOps` — all git and gh shell-outs), `Repositories` (SQLite), `SlackClient`. **All wiring happens only in `startDaemon()` of `src/daemon/server.ts`.** Assemble new dependencies there; the core only receives them as interfaces.

### Master vs Worker — the spine of this codebase (don't confuse them)
The input models are **different.**
- **Master** (`src/core/master-agent.ts`): `runTurn(text)` → one **string prompt** to `query()` + SDK `resume: sdk_session_id` to continue context. Turns are **serialized** in a per-session `turnChain` (concurrent sends are queued, not interleaved). `buildSystemPrompt()` = `claude_code` preset + the 10 most recent memories + repo catalog injection. Every turn it receives the base in-process MCP servers (memory/repos/fleet) plus a per-source `capabilities()` overlay (see Tools).
- **Worker** (`src/core/worker.ts`): passes a **`MessageQueue` (streaming input)** to `query()` to keep the session alive → can follow up via `send()`. **No MCP servers** — it just runs the default `claude_code` toolset (Bash/Edit, etc.) with `cwd: worktree`. ⚠️ **The user echo of a mid-turn `send` is recorded not the moment it is sent but at the next `result` (turn boundary)** (`deferred` FIFO) — because the `MessageQueue` only hands the message to the SDK on the next turn, the echo is deferred to the boundary so it doesn't land in the middle of an in-progress turn's output (parity with the master `turnChain` deferral). In the meantime the desktop shows a `pendingByWorker` "waiting" bubble, then reconciles it with the echo (`clientMsgId`).
- **Masters can also run on Codex** (`session.create provider:"codex"`; desktop New Session page has a claude/codex selector + a `codexMasterModel` picker — see the codex model picker note below): each turn spawns a **per-turn ephemeral `codex app-server` child** and calls `thread/start`/`thread/resume` with **fresh `developerInstructions` every turn** (= the per-turn `buildSystemPrompt()` injection Codex otherwise lacks). Tools reach the child over the daemon's **MCP bridge** (`/mcp/<session-token>` mounted on the daemon http port; `src/daemon/mcp-bridge.ts`), not in-process MCP servers. ⚠️ **bypassPermissions only** — restricted sandboxes silently swallow the bridge's turn-scoped MCP calls (`item/started` fires, `tools/call` never arrives, the turn hangs); non-bypass permission modes are rejected at turn start. Slack threads can run on codex too via the `slackProvider` setting (`"claude"|"codex"`, default claude, opt-in) — same bypassPermissions-only constraint applies. **Fixed (P2.5): the bridge token no longer rides argv** — it used to be visible in the child's argv (`-c mcp_servers.rookery.url=...`, readable via `/proc/*/cmdline` by other local users on multi-user Linux); now each codex master gets a per-session `CODEX_HOME` (`src/daemon/codex-home.ts`) under `<rookery home>/codex-homes/<sessionKey>/`, and the URL lives only in that dir's `config.toml` (mode 0600) — the child spawns with the `CODEX_HOME` env var and no `-c` arg. Rollouts and `auth.json` live in that same per-session dir: `auth.json` is a symlink to `~/.codex/auth.json` when no in-app `codexApiKey` is set, or provisioned in-place (`account/login/start`) when one is; the dir is removed on `session.delete` (best-effort). The per-turn **inactivity watchdog** is opt-in (`codexTurnIdleTimeoutMs` setting, default `0` = disabled; positive ms enables it): when enabled, it interrupts then kills a turn that goes totally silent (timer resets on any event) and fails it with a `notice.codexTurnTimeout` notice. A separate **handshake/thread-start timeout** (`codexHandshakeTimeoutMs` setting, default 30000ms, `0` disables) covers the pre-turn phase the idle watchdog can't reach: a child wedged during spawn/`initialize`/`thread/start` never reaches the turn where the idle watchdog arms, so this timeout races the whole handshake phase and fails the stream terminally on expiry. Orphan `codex-homes/<id>/` dirs (left by a session-delete crash or a fork whose `createSession` throws after seeding) are **swept at boot** (`gcOrphanCodexHomes`, best-effort, runs after `fleet.rehydrate()`), so they can't leak permanently. Per-session cleanup (MCP-bridge release + `CODEX_HOME` removal) is owned by a single caller, `SessionManager.delete` (fires exactly once via an injected `onSessionDelete` hook — no separate Connection-side call). Slack (`slackProvider`) and automation (`provider`) codex origins are UI-selectable now (Settings page Slack section / Automation form), not env/DB-only. See `docs/2026-07-06-p2-codex-master.md`, `docs/2026-07-06-p25-codex-hardening.md`, and `docs/2026-07-06-p3r-codex-hardening-finish.md`. **Codex master fork now works (P3):** `SessionManager.fork` runs `thread/fork` in the source session's per-session `CODEX_HOME`, then seeds the new session's `CODEX_HOME` by copying the source's `sessions/` rollout tree (the forked rollout is a parent-referencing delta, so the whole tree must come along, not just the one file) — context is preserved (live-verified: the fork recalled a fact stored in the source session). Codex **worker** fork is unchanged (workers run on the shared home, not a per-session one). See `docs/2026-07-06-p3-codex-fork-automation.md`.

- **Codex model picker** (desktop): the Codex model/effort fields (WorkerSpawnModal, NewSessionPage, AutomationForm, SettingsPage `codexWorkerModel`/`codexMasterModel` defaults) are **dropdowns driven by the app-server `model/list` catalog**, not free text. The daemon fetches it via `codex.models.list` → `src/core/codex-models-provider.ts` (a short-lived `codex app-server` child: `initialize`→`model/list`, mapped to `CodexModelInfo{ id, displayName, defaultEffort, supportedEfforts, isDefault }`, **cached first-success only**; any failure → `null`, not cached, so post-boot auth recovers). The desktop caches it in the `codexModels` store slot; each model's `supportedReasoningEfforts` drives a **per-model effort selector** (picking a model pre-selects its `defaultReasoningEffort`). ⚠️ When the catalog can't be fetched (codex missing/unauthed) `codexModels` is `null` → the surfaces **fall back to the free-text model input** (today's behavior, no breakage). A leading `""` option = "use the `codexWorkerModel`/`codexMasterModel` settings default" (never auto-picks a catalog id). See `docs/superpowers/specs/2026-07-06-codex-model-picker-design.md`.

Model/effort are injected **not as values but as resolvers (`() => string`)**, so they are **re-evaluated every turn** → `Settings` changes are reflected immediately in live sessions. Snapshotting them as strings at session-creation time breaks this behavior.

### Fleet = the control plane (worktree isolation)
`src/core/fleet-orchestrator.ts`. **One** `FleetOrchestrator` per daemon (a global pool) is injected into all sessions → from any session you can list/inspect/control the entire fleet.
- spawn → create a worktree + branch `rookery/<id>` at `~/.rookery/worktrees/<id>` (`git worktree add -b … <base>`) → start the worker.
- Workers can run on either backend: `spawn_worker`/`worker.spawn` take `provider: "claude" | "codex"` (default claude); codex workers run via a per-worker `codex app-server` child (`src/core/codex/`, curated protocol types ground-truthed against CLI 0.144.1 — regenerate with `codex app-server generate-ts` on bumps; decode is tolerant so older 0.142.x servers still work). Masters can now run on Codex too (see the Master vs Worker section above).
- ⚠️ **The orchestrator manages only worker lifecycle. Commit/push/PR is done directly by the worker in its own worktree via bash (`git`/`gh`)** (the master instructs it via `send_worker`). **There is no automatic PR pipeline** — `GitOps` does only worktree/diff/checkpoint/fetch (no commit/push/openPr). The README §Fleet description of an auto commit→push→`gh pr create` pipeline is **stale**; trust the source.
- `waitUntilSettled()` resolves **only on termination (stopped/done/error)** — **it does not resolve on idle**. `flows` is used to **wait for the termination drain (close→waitAllSettled)** (not for concurrency-cap accounting). **There is no concurrent worker cap — spawns are never rejected.** A lifetime **USD cost budget** (`costBudgetUsd`, default OFF) now stops a worker / warns a master once `cumCostUsd >= budget`, set per-spawn (`spawn_worker`/`fleet.spawn`) / per-automation / via the `workerCostBudgetUsd` settings default; overshoot is bounded to ≤ one turn (checked at turn boundaries). A lifetime **turn** cap is still future.
- **Async notification:** `spawn_worker`/`send_worker` accept `notify:true`. `WorkerNotifier` (`src/core/worker-notifier.ts`) arms the requesting master to be woken when that worker next *settles* (idle on success, or a terminal failure — so the master isn't left waiting forever on a failed dispatch). Pending arms persist in the `pending_notifications` table so they survive a restart.
- `stop_worker` (keeps the worktree) vs `discard_worker` (removes the worktree + branch, losing uncommitted work).
- Restart: on daemon boot `fleet.rehydrate()` restores detached entries from DB rows (diff/discard/stop keep working). If both the worktree and `sdk_session_id` are alive, lazy resume (→idle); otherwise `orphaned`. The live streaming conversation itself dies with the process and does not come back.

Worker state union: `running | idle | background | stopped | done | error` + the orchestrator-only DB states `failed`/`orphaned`. Live states are **derived** (`Worker.reconcile()`: `turnActive ? running : bgTasks>0 ? background : idle` — docs/superpowers/specs/2026-07-11-worker-state-graph-design.md): **`background`** = turn ended but harness-tracked background tasks (run_in_background shells etc., claude-only — codex has no bg concept) still run; **`idle` now truthfully means ALL assigned work complete**. The SDK **auto-wakes** the model when a bg task settles (live-verified) → a spontaneous `idle/background → running` edge with no send(); `interrupt` does NOT kill bg tasks (only `stop` does, via the subprocess tree). A **settle-grace** (`WorkerDeps.settleGraceMs`, default 3000) holds `background` between the last settle and the wake so NO transient idle is ever emitted (event-driven consumers would fire on even a 1ms blip — live-observed ~4s on worker 74022a19); the wake turn's `init` frame resolves the grace near-instantly (grace-scoped only — a boot-time init must not flip a quiescent worker), and expiry = no wake → truthful idle. `WorkerNotifier` deliberately excludes `background` from SETTLED (no premature master wake). **`done` is retired from live writes** (natural stream end → notice + `stopped`); it survives in the union only for legacy rows.

### Tools = in-process MCP servers (master only)
`src/tools/`. Each file exports raw **`*ToolDefs(deps)`** (structurally `SdkMcpToolDefinition[]` — name/description/zod raw shape/handler) shared by both providers, plus `create*ToolsServer(deps) = createSdkMcpServer({ tools: *ToolDefs(deps) })` for Claude. The base set injected every master turn is **memory / repos / fleet**; `schedule` and per-source servers (e.g. `slack-thread`) are layered on via the turn's `capabilities()` overlay. `MasterAgent` passes these groups as `toolDefs` (`Record<serverName, ProviderToolDef[]>`) on the `AgentBackend` port: `ClaudeBackend.startTurn` wraps each group with `createSdkMcpServer` and merges into the SDK's `mcpServers` (byte-identical Claude path); `CodexBackend.startTurn` instead registers/refreshes them on the daemon's MCP bridge for that session, so the per-turn child gets fresh closures each turn with no update API needed. `AskUserQuestion` travels as a def (`src/tools/ask-user-question-def.ts`) so a codex master gets the SAME interaction-registry-backed Q&A flow — the Claude adapter deliberately drops that def since Claude already has AskUserQuestion natively (avoids double-exposing it). The `*_TOOL_NAMES` sync rule is unchanged (still the exposure gate for `allowedTools`).
- **memory** (`memory-tools.ts`): `remember`, `recall`
- **repos** (`repo-tools.ts`): `register_repo`, `list_repos`, `update_repo`, `remove_repo`
- **fleet** (`fleet-tools.ts`): `spawn_worker`, `send_worker`, `interrupt_worker`, `list_workers`, `get_worker_status`, `view_worker_transcript`, `view_worker_diff`, `stop_worker`, `discard_worker` (the master is purely async — `spawn_worker`/`send_worker` take `notify:true` to be woken on completion/failure; there is no blocking `await_worker`. `send_worker` does **not** interrupt a mid-turn worker — it queues to the next turn boundary; `interrupt_worker` aborts the current turn but keeps the session/worktree, so a redirect = `interrupt_worker` → `send_worker`, vs `stop_worker`/`discard_worker` which are terminal)
- **schedule** (`schedule-tools.ts`): `schedule_wakeup`, `schedule_list`, `schedule_cancel` — the master scheduling its own future wake-ups (the `once` automation kind)
- **slack-thread** (`slack-thread-tools.ts`): `read_thread` — injected only for Slack-origin sessions, lets the master read the current thread transcript

Because `permissionMode:"bypassPermissions"`, the only exposure gate is the `allowedTools` allowlist → you **must keep** each file's `*_TOOL_NAMES` constants (format `mcp__<server>__<name>`) **in sync** with the actual `tool()` names. If they drift, the tool registers but silently can't be used. **Workers do not receive these tools** (they cannot spawn fleet or access memory themselves). Recent memories are injected into the system prompt every turn automatically, separate from any `recall` call.

### External MCP server (rookery-as-MCP)
Exposes fleet control to **external MCP clients** (Claude Code/Cursor/Codex) — `claude mcp add rookery --transport http http://127.0.0.1:8787/mcp-ext/<token>`. It is a **SECOND `McpBridge` instance** (`new McpBridge({ basePath: "/mcp-ext" })`) mounted on the same daemon http server, owned by `ExternalMcpController` (`src/daemon/external-mcp-controller.ts`) — token/session namespace-isolated from the codex `/mcp` bridge (see docs/2026-07-11-external-mcp-server-design.md). **Off by default (fail-closed)**, gated by the `mcpExposure` setting (`off`/`readonly`/`full`, unknown value → off). `readonly` = observation only (`list_workers`/`get_worker_status`/`view_worker_transcript`/`view_worker_diff`/`list_repos`); `full` adds `spawn_worker`/`send_worker`/`interrupt_worker`/`stop_worker`/`discard_worker`. The tool set IS the gate — there is no `allowedTools` here; `readonly` simply doesn't register the mutating tools. Tools live in `src/tools/external-tools.ts` (reuse `spawnWorkerImpl`/`formatTranscript`; `notify` is dropped — meaningless for the container home session; external spawns are attributed to the hidden `external:fleet` home session so they show in fleet views). Auth = a persisted 0600 `~/.rookery/mcp-token` (`fixedToken` on `ensureSession` pins the URL across restarts; `mcp.regenerate_token` rotates it via `release()`+re-register). `reconcile()` runs at boot and on every `settings.set` that touches `mcpExposure`; `mcp.status`/`mcp.regenerate_token` protocol messages carry the computed URL. ⚠️ `full` lets an external agent drive the bypassPermissions fleet — the desktop shows a warning; the `workerCostBudgetUsd` default still applies to external spawns.

### Persistence (SQLite)
`src/persistence/`. `better-sqlite3` (synchronous), `WAL` + `foreign_keys=ON` + **`STRICT` tables**. Migrations are an **append-only `MIGRATIONS` array in `db.ts`** (index = version, forward-only, one transaction). **Never modify existing entries; only add new ones** — `db.test.ts` asserts the applied schema version stays `=== MIGRATIONS.length`, and a DB newer than the build is rejected at open. Tables: `sessions`, `messages` (text + last_activity), `session_events` (master transcript events — on restart/reconnect even tool/thinking/metrics/notice are restored; the master counterpart of `worker_events`), `workers`, `worker_events`, `worker_checkpoints`, `memories`, `repos`, `settings`, `automations`, `pending_notifications`, plus Capability Center's `capability_packs`, `capability_bindings`, `capability_trust`, and `capability_secrets`. All access goes through `Repositories`.

### Capability Center (managed Claude runtime)
`src/core/capabilities/`. `manifest.ts` strictly validates a local `capability.json`, referenced instruction files, skill roots/frontmatter, every contained file/symlink, public-vs-secret MCP fields, and bounded whole-pack digest. `registry.ts` owns persistence-backed registration, refresh/change metadata, exact-digest trust, declared write-only secrets, and sanitized Library projections. `resolver.ts` chooses one binding per pack with precedence `worker > session > repo-local > repo-shared > rookery`, audience-filters by agent/origin, and produces a deterministic secret-free runtime projection. `runtime-state.ts` tracks desired/applied drift in memory; `service.ts` merges it into authoritative snapshots. Mutations emit `capabilities.changed`; application emits `capabilities.runtime`.

**Slice 3 applies managed packs to Claude only.** `src/core/claude-capabilities.ts` lowers instructions, skills, and MCP specs; `src/daemon/capability-runtime.ts` copies and revalidates trusted bytes into an immutable revision, generates local plugins, and resolves write-only values only into the Claude child environment. Masters resolve immediately before every `startTurn` (changes apply next turn). Workers resolve once before initial/lazy-resumed `openSession`; later changes show `pending-reload` and do not restart the live stream. Codex application, worker hot reload/GC, repository-shared discovery, and managed command actions remain later slices. Required missing state blocks before provider spawn; optional MCP refs are unavailable; Side MCP remains suppressed. Protocol, events, logs, revisions, generated files, tests, and UI must never receive secret values. The example under `docs/examples/capability-pack/` is validated by `test/core/capabilities/example-pack.test.ts`.

### Daemon / Protocol
`src/daemon/`, `src/protocol/messages.ts`. http `/health` + WebSocket `/ws` in `noServer` mode. Single-instance is a **PID file lock** (`~/.rookery/daemon.pid`, exclusive `wx` creation; stale ones are judged via `process.kill(pid,0)` and cleaned up). WS auth: `~/.rookery/ws-token` (0600 shared secret) + an **Origin gate** (rejects non-local Origins) + `timingSafeEqual`. A non-local bind (`ROOKERY_HOST`) sends the token in plaintext, so it only raises a stderr warning. On boot, `src/daemon/fs-hardening.ts` tightens `~/.rookery` to `0700` and sensitive files to `0600` (best-effort, never throws).
Protocol: **inbound is validated with the zod `clientMessageSchema` (discriminated union)**, outbound `ServerMessage` is just `JSON.stringify`'d (no schema validation). `reqId` correlates request and response (some fire-and-forget messages have no reqId). Events flow over `EventBus` channels: session id, `FLEET_CHANNEL("@fleet")`, `ALL_CHANNEL("@all")`. When subscribing to `@all`, duplicate per-session subscriptions are deduped, and `EventBus.deliver` wraps each listener in try/catch so one bad listener can't break fan-out.

### Localization (i18n)
Two independent, zero-dependency catalogs; **Korean is the default** everywhere.
- **Daemon side — `src/core/i18n.ts`** (`t(locale, key, params?)`, `resolveLocale`, `DEFAULT_LOCALE="ko"`): ko/en strings for Slack/CLI output and core notices (`notice.*` / `slack.*` / `cli.*` / `interaction.*`). The Slack adapter resolves its locale live per message from the `slackLocale` setting; the CLI resolves from env (`LC_ALL`/`LANG`).
- **Structured core notices:** `classifySystemPush` returns `{code, params, text}` where `text` is pre-rendered at `DEFAULT_LOCALE`, and the master attaches `code`+`params` to `master.notice` events so each client can re-localize. `worker.ts` consumes the pre-rendered `text` and is deliberately untouched by this.
- **Desktop side** has its own renderer catalog (`apps/desktop/src/renderer/i18n/`, ko/en, runtime-switchable) plus a separate main-process catalog (`apps/desktop/src/main/i18n.ts`). ⚠️ The renderer `notice.*` keys/param-names must stay byte-identical to the daemon `notice.*` keys (intentional cross-build duplication).

### Slack adapter
`src/slack/`. Only when **both** the bot and app tokens are present does the daemon also bring up a bolt Socket Mode bot (otherwise it is silently disabled and the daemon is fine). **Token, cwd, allowlist, allow_all, refusal message/reply, and locale are all resolved from `Settings` (DB)** — tokens prefer the DB with an env fallback, the rest are settings-only. Instead of a `config`, the slack layer receives a `slackConfig: () => {...}` resolver and **resolves on every call (live)**: tokens at connect time, allowlist/cwd/locale per message. **Thread = session** (external key `slack:team:channel:threadTs`, persisted → the same conversation survives a daemon restart). The allowlist is **fail-closed**: if the allowlist is empty and allow_all is not set either, **everyone is rejected** (stderr warning at connect); a non-permitted user gets the (configurable) `DEFAULT_SLACK_REFUSAL` reply. Responses stream as plan cards (`task_display_mode:"plan"`), and the master's tool calls are rendered as in_progress→complete tasks.

Bolt **boots asynchronously** (it does not block startup): `SlackController` (`src/slack/controller.ts`) is the single owner of lifecycle + state (`unconfigured` (no token) / `off` (toggle off) / `connecting` / `up` / `error`), and on every state transition emits a `slack.status` CoreEvent to `sessionId:ALL_CHANNEL` to broadcast to all clients. server.ts brings it up with `void slack.boot()` and injects it into the Connection. **Turning it on/off is the `slack.set {enabled}` command** → `controller.setEnabled` → the persisted setting `slackEnabled` ("1"/"0", default on) + start/stop. **When a token changes via `settings.set`, the Connection calls `controller.reconcile()`** → it disconnects, re-evaluates against the current configured/enabled, and (re)connects immediately on save (`configured` is a resolver because tokens change at runtime). On connect, the Connection replies once with the current state in `events.subscribe` (initial sync). The UI (desktop) exposes it via a bottom-left indicator + the Slack section of the settings page (status/toggle + token/cwd/allowlist/allow_all/refusal/locale fields).

### Automation (Triggers + Actions) — cron + Slack events
An automation rule = a **trigger** (`cron` | `interval` | `slack` | `worker`) + an **action** (`master` | `worker`). The `automations` table (STRICT): `trigger_type`/`trigger_config_json` (cron: `{cron,timezone}` · interval: `{everyMinutes}` · slack: `{channels?,keyword?,fromUsers?}` · worker: `{repo?,on?,label?}`) + `action_type`/`action_config_json` (master: `{prompt,cwd,sessionMode}` · worker: `{repo,task,base?}`) + `model`/`effort`/`permission_mode`/`max_turns`/`next_run_at`/`last_*`. The types are `Automation`/`AutomationTrigger`/`AutomationAction` (`kind` discriminated union), CRUD via `Repositories`. (`interval` is a recurring time trigger — "every N minutes", forward-from-now — sharing cron's Scheduler/overlap path; `everyMinutes` round-trips through the generic `{kind,...rest}` serialization, no migration.)

- **`AutomationDispatcher` (`src/core/automation-dispatcher.ts`) = the single firing point.** Every trigger source calls `run(a, vars)` → **only recurring time triggers (cron/interval)** skip overlap via a per-id **in-flight Set** (`last_status:"skipped"`, preventing schedule pile-up); **slack (event) triggers allow concurrent execution** (processed per message, no drops) → `runAutomationAction` → `setAutomationRun` (last_run/status/error, preserving `next_run_at`) → emit `automation.changed`. The source only decides "what, with which vars." (For `once`, the Scheduler claims by nulling `next_run_at` before firing — the tick skips claimed rows — and deletes after the run settles; a crash mid-run is re-armed at boot and refires, at-least-once.)
- **`runAutomationAction` (`src/core/automation-action.ts`)** = a pure action. It substitutes `{{message}}`·`{{channel}}`·`{{user}}`·`{{ts}}`·`{{threadTs}}` (falls back to the message ts if there is no thread)·`{{team}}` into the master prompt / worker task (function-form replacer, `$`-safe; empty string when missing) → master `runTurn` (reuse[`automation:<id>`]/fresh) / worker `fleet.spawn` (the hidden home session `AUTOMATION_FLEET_SESSION_KEY="automation:fleet"`). ⚠️ **Untrusted Slack vars are fenced**: each substituted value is wrapped in `<untrusted-… id="<nonce>">…</untrusted-…>` with a fresh per-call nonce, and the value is neutralized (nonce stripped + a Zero-Width Space inserted into any literal `untrusted-` tag) so injected text can't spoof the fence.
- **Trigger source ① cron/interval — `Scheduler` (`src/core/scheduler.ts`, one per daemon):** uses `croner` to **only compute the next run** (no internal timer → deterministic via an injected `now()`); an injected tick selects `(cron|interval) && enabled && next_run_at<=now` and **advances next_run first → re-reads fresh → `dispatcher.run(a,{})`** (no double recording — recording is the dispatcher's). **`interval` next-run = `now + everyMinutes`** (forward-from-now, computed at fire time — no wall-clock anchoring, no catch-up burst if the daemon was down; min 1 min, above the 30s tick). `runNow` fires once immediately (does not advance next_run), **enabled defaults off**, **does not backfill**, and ignores slack rules.
- **Trigger source ③ worker-settled — `src/core/worker-trigger-source.ts`** (docs/superpowers/specs/2026-07-11-worker-settled-trigger-design.md): subscribes `worker.status` on `@all` (the WorkerNotifier signal, truthful since the background-aware state machine) and fires `worker`-kind rules when a fleet worker settles. `on` buckets: `idle` (dispatch complete) / `stopped` (intentional end; legacy done folds in) / `failure` (error·failed·orphaned) — **absent/empty → `["stopped","failure"]`, idle is opt-in** (most re-fire-prone). Two guards: **self-loop** (workers homed in `automation:fleet` never match — a worker-trigger + worker-action rule can't react to its own spawns) and an **in-memory once-per-(automation, worker) latch** (idle recurs per dispatch; terminal statuses double-emit via the two status writers; restart resets the latch — at-most-once per process). Template vars `{{workerId}}/{{repo}}/{{branch}}/{{status}}/{{label}}/{{tail}}` (fenced like slack vars; tail = last assistant message, shared `extractWorkerTail` with the notifier). Concurrent dispatch (event trigger; the overlap-skip stays cron/interval-only). Recipe: "구현 워커 stopped → 같은 `{{branch}}`에 리뷰어 워커 스폰" = worker→worker handoff.
- **Trigger source ② slack — `src/slack/trigger-source.ts` + `app.message`:** filters Bolt message events via `src/slack/message-text.ts` — it excludes only **our own bot (its own `bot_id`, looked up via `auth.test` at connect)** and noise subtypes like edit/delete, while **letting other bots/integrations (CI, monitoring, alerts) through** (the `bot_message` subtype is allowed; feedback loops are blocked by self-exclusion). `extractSlackText` **melts Block Kit blocks/attachments/rich_text into text** for matching and `{{message}}` (rich_text mirrors m.text so it's skipped as a duplicate; fallback is used only when there's nothing else). Then `makeSlackTriggerHandler` evaluates `enabled && trigger.kind==="slack"` rules with `matchesSlack` (`src/core/automation-match.ts`, pure: channel / keyword [case-insensitive substring] / sender, all matching when empty) → on a match, `dispatcher.run(a,{message,channel,user})`. Independent of the conversational (assistant/mention) path.
- Protocol `automation.*` (connection→`AutomationProvider`, wired in server.ts; create/update/set_enabled call `scheduler.reconcile`); for cron triggers, `automationInputSchema.superRefine` rejects saving an invalid cron via `isValidCron`. On change, `automation.changed` is broadcast to `ALL_CHANNEL` → the desktop `AutomationPage`/`AutomationModal` refetches.
- ⚠️ **Slack app prerequisites (user configuration, the bot can't do it):** subscribe to the `message.channels` event + the `channels:history` scope + invite the bot to the target channels. Without these, message events are not received.
- **Codex-origin automations (P3):** an automation (master or worker action) can run on codex via the `provider` field on the automation row (`"claude"|"codex"`, default claude) — the desktop `AutomationForm` exposes the same selector as worker-spawn. Codex masters are **bypassPermissions-only** (see the Master vs Worker section), so a codex automation configured with a non-bypass `permission_mode` fails its run with a clear error surfaced in `last_error`/status (automations default to bypass, so this only bites a mis-config). Codex workers are unaffected — workers aren't bypass-guarded. `automation.provider` is independent of the Slack adapter's `slackProvider` setting: a Slack-triggered automation still runs on whatever `automation.provider` says, not on `slackProvider`. ⚠️ Changing an automation's provider on a **reuse-mode master** automation takes effect only after its `automation:<id>` session is deleted — the session's backend is fixed at creation (`getOrCreateByKey` applies `provider` only on first creation of that key); fresh-mode masters and worker actions apply the new provider immediately (a new session/worker is created each run).
- ⚠️ **Prompt injection / unattended runs:** untrusted channel text enters the master prompt as fenced `{{message}}` (see above), but because automation runs `bypassPermissions` and unattended, budget guards remain partial: a lifetime USD cost budget (`costBudgetUsd`, default off) now guards runaway cost (per-spawn / per-automation / `workerCostBudgetUsd` default), but a lifetime **turn** budget is still absent beyond the per-action `max_turns`. `canUseTool` auto-allows (an unattended turn isn't blocked by AskUserQuestion).
- **Extension seam:** a new trigger kind (webhook/reaction/interval) only needs a union member + config + (event-type) a matcher or (time-type) next-run computation + a source file + server.ts wiring — the dispatcher/execution/model/Scheduler are unchanged.

### `~/.rookery` home layout (`ROOKERY_HOME`)
`rookery.db` (WAL; Capability Center registry/trust/bindings/local secrets also live here) · `daemon.pid` (single-instance lock) · `ws-token` (0600 WS auth) · `mcp-token` (0600 shared secret for the External MCP server URL `/mcp-ext/<token>` — separate from ws-token so it rotates independently) · `daemon.log` (daemon stdout/stderr — Node version/auth errors are logged here) · `worktrees/<id>/` (per-worker git worktree) · `capability-runtime/<revision>/` (immutable copied sources + generated Claude local plugins; directories 0700, metadata 0600, no secret values) · `slack-files/` (downloaded Slack attachments). Registered pack sources remain in place; a trusted revision is copied lazily only when a Claude target launches. It's outside the repo, so it's not subject to `.gitignore`; the home dir is hardened to `0700` on boot.

## Fragile conventions / pitfalls

- **Node 22 ABI** — see the "Required" section above. The #1 pitfall of all.
- **ESM NodeNext** — relative imports **require the `.js` extension**, type-only uses need **`import type`** (`verbatimModuleSyntax:true`). `rootDir: src`.
- Don't confuse the **master (string prompt + resume) vs worker (MessageQueue streaming)** input models.
- **Model/effort are re-evaluated every turn via a resolver** — snapshotting them means runtime `Settings` changes aren't reflected.
- **Control-plane model** — there is no automatic PR pipeline. The worker opens PRs directly. README §Fleet is stale on that part, so **trust the source.**
- **`bypassPermissions` applies to both master and worker**, and fleet fan-out multiplies the blast radius. A lifetime USD cost budget (`costBudgetUsd`, default off) now exists — opt in per-spawn/per-automation or via the `workerCostBudgetUsd` settings default — but a lifetime **turn** cap is still absent (only per-automation `max_turns`), and the cost guard is opt-in, so run it only in trusted environments/paths.
- **Migrations are append-only** — never modify existing entries.
- **There are two terminal-state writers** (`Worker.transition` vs `FleetOrchestrator.setStatus`), but **the terminal write-once is enforced at the single chokepoint `repos.setWorkerStatus(id, status, force?)`** — once terminated (`stopped/done/error/failed/orphaned`) it can't be overwritten with another value (force = only user stop/discard and rehydrate are exceptions). Adding a new writer is safe as long as it goes through this guard.
- **Passing effort to Haiku is an API 400** — always gate with `effortApplies(model) && coerceEffort(effort)`.
- **i18n** — Korean is the default. New daemon-side user-facing strings (Slack/CLI/notices) go through `src/core/i18n.ts`; new `notice.*` codes must be added to BOTH the daemon catalog and the desktop renderer catalog with matching param names. Code comments are written in English.
- **Codex worker auth** — an in-app `codexApiKey` setting now EXISTS. When set, codex workers run under `<rookery home>/codex-home` (`CODEX_HOME` redirection + one-time `account/login/start` provisioning); **toggling the key strands pre-toggle thread rollouts in the other home** (resume of old workers fails cleanly until the key state is restored). When unset, workers fall back to the user's `~/.codex/auth.json` (`codex login`, P1 behavior). Either way, the app-server child does NOT read the `CODEX_API_KEY` env var (verified against rust-v0.142.5).
- **Root gates don't cover `apps/desktop`** — any change to shared daemon types the renderer consumes (`SettingsValues`, `WorkerRow`, `notice.*` codes) must also run `npm -w apps/desktop run typecheck && npm -w apps/desktop test` (P1 broke desktop typecheck this way; fixed in 986da1b).

## Testing

- Tests mirror `src/**` 1:1 in `test/**`. Thanks to the DI ports, unit tests run **without a real SDK, git, or network**.
- **`test/helpers/fake-query.ts` is the canonical way to mock the SDK `query()`** (`fakeQuery(script)`). To verify `query()` options (model/effort/systemPrompt/permissionMode), wrap the fake in a capturing closure (see `capture()` in `test/core/master-agent.test.ts`). `fakeBackend`/`fakeStreamingBackend` (fake-query driven through the real `ClaudeBackend`) is how Worker/Master tests inject the port; raw `fakeQuery` remains for adapter/aux tests.
  - ⚠️ **`fakeQuery` is finite** (end of script → generator terminates → the worker lands `stopped` with a "Stream ended" notice; `done` is retired). **A real streaming SDK iterator only ends when the input is closed.** This gap hides happy-path lifecycle issues, so don't overtrust green tests as end-to-end evidence.
- The DB is `openDb(":memory:")` + `new Repositories(db, now?)` (deterministic timestamps via an injected clock). git is the injected `FakeGitOps` (records calls); `RealGitOps` is verified separately against a temp repo (`fs.mkdtempSync`).
- Background fleet flows are awaited with `FleetOrchestrator.waitAllSettled()`.
