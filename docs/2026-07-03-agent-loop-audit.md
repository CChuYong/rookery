## 1. [HIGH] Full reload wipes the replayed pending interaction card: session.history seed drops it — approval turn hangs invisibly again

**Where**: `apps/desktop/src/renderer/store/reduce.ts:68` · verified 3/3 lenses

Commit 7c080ba makes the daemon replay pending interaction.request events on events.subscribe (connection.ts:272) so a full desktop reload doesn't leave the held canUseTool turn hung. But on a fresh reload the replayed card is destroyed moments later by seedSessionLog. Timeline: (1) onOpen sends events.subscribe → the daemon replays the pending card → reduceEvent appends it to the EMPTY logsBySession[sid] (one interaction item, zero message items). (2) The view-restore effect (App.tsx:326-330) / select() requests session.history and calls seedHistory. (3) seedSessionLog anchors prev against committed by counting kind==='message' items; prev = [interaction card] contains zero messages while the persisted transcript contains >=1, so the anchor loop never matches, cut stays -1, and the function returns `committed` — dropping the card. Interaction events are never persisted to session_events (intentional), so committed can never contain it, and nothing ever re-requests pendingEvents afterwards. TCP ordering guarantees the replay frame arrives before the session.history reply, so the wipe is deterministic. Result: the master turn is still blocked in InteractionRegistry.request, running pulses forever, but no card is visible and there is no way to answer — the exact pre-7c080ba hang. Verified empirically by running the real reducer: replay-then-seed on an empty log yields 0 interaction cards. The daemon-side test only asserts the card is replayed over the wire; no test covers it surviving the subsequent seed.

**Trigger**: 1) In a master session with permissionMode 'default' (or AskUserQuestion), the master turn raises an approval card and blocks on canUseTool. 2) Fully reload / relaunch the desktop app while the card is pending — the detached daemon keeps the turn blocked. 3) On reconnect the daemon replays the card (visible for a frame at most), then the restored session's session.history seed replaces the log and the card vanishes. 4) The UI shows the session 'running' with a normal transcript, no card, and no way to respond (the requestId is no longer displayed); only Stop aborts the turn.

## 2. [HIGH] httpServer.listen has no 'error' handler: port-bind failure leaves a zombie daemon permanently holding the PID lock

**Where**: `src/daemon/server.ts:313` · verified 3/3 lenses

startDaemon acquires the single-instance PID lock and opens the DB, then awaits a Promise that only resolves via the listen success callback. No 'error' listener is registered on httpServer. If bind fails (EADDRINUSE), Node emits 'error' with no listener, which throws asynchronously and is swallowed by installProcessGuards' uncaughtException log-then-survive handler (src/index.ts:43-46). Result: startDaemon never resolves nor rejects, main() hangs forever, the process stays alive holding daemon.pid. Every subsequent daemon start throws 'rookery daemon already running (pid X)' while /health stays down, so the CLI's ensureDaemon and the desktop's DaemonManager fail with 'daemon did not become healthy' until the user manually kills the zombie. lock.release() and db.close() also never run.

**Trigger**: 1) Some other dev process listens on 127.0.0.1:8787. 2) User runs `node dist/index.js` (CLI) — pingHealth fails, ensureDaemon spawns a detached daemon child. 3) The child acquires the PID lock, opens the DB, then listen() fails with EADDRINUSE → unhandled 'error' → uncaughtException guard logs it and the process survives, parked at the never-settling await. 4) ensureDaemon times out after 5s. 5) Every retry (CLI or desktop) spawns a new child that dies at acquireSingleInstance('already running') because the zombie's pid is alive — permanent wedge until the zombie is SIGKILLed.

## 3. [HIGH] Dispatcher finally-write rewinds next_run_at to a stale snapshot — overlap-skipped cron automations refire immediately and run back-to-back forever

**Where**: `src/core/automation-dispatcher.ts:42` · verified 3/3 lenses

AutomationDispatcher.run snapshots the automation row at fire time and its finally block unconditionally writes all four columns back, including nextRunAt from that snapshot (`setAutomationRun` is a blind UPDATE). The docs say the run-record 'always preserv[es] next_run_at (the Scheduler owns that)' — but the code preserves the fire-time value, not the current DB value. Any writer that changes next_run_at while a run is in flight is silently reverted when the run completes: (a) Scheduler.fireCron advances next_run BEFORE dispatching even when the dispatch is then overlap-skipped, so during a run longer than the cron period each skipped tick advances next_run_at, and the long run's finally rewinds it to a now-past timestamp → the next 30s tick fires immediately. With run duration > period this repeats after every completed run, so the automation executes back-to-back with zero idle gap forever — exactly the pile-up the in-flight guard exists to prevent, in an unattended bypassPermissions context with no budget guards. (b) automationProvider.update/setEnabled → scheduler.reconcile writing a new next_run_at mid-run is likewise clobbered, so a schedule edit made during a run is silently reverted. Secondary: the overlap-skip path also overwrites the in-flight run's transient 'running' status with 'skipped'. No test covers next_run_at across an overlap.

**Trigger**: 1) Create an enabled cron automation `*/5 * * * *` whose master action takes ~12 min. 2) T0 tick: fireCron advances next_run_at to T5, dispatcher starts the run (inflight). 3) T5 tick: advance to T10, 'skipped'. 4) T10 tick: advance to T15, 'skipped'. 5) T12: the run finishes; dispatcher finally writes nextRunAt = T5 (fire-time snapshot) → next_run_at is now in the past. 6) T12:30 tick: fires immediately, advances to T15, runs another 12 min → same rewind at completion → the automation runs continuously back-to-back with no scheduled gap. Variant: edit the schedule in the UI while a cron run is in flight; when the run completes, finally restores the old next_run_at and the edit is silently lost until the next fire.

## 6. [HIGH] restore() TOCTOU: send_worker/worker.send can start a turn while git checkout is rewriting the worktree

**Where**: `src/core/fleet-orchestrator.ts:427` · verified 3/3 lenses

restore() checks the running guard exactly once, then awaits git.restoreCheckpoint (a `git checkout <sha> -- .` subprocess that takes 100ms–seconds on a large tree) while holding no lock and setting no flag. During that await, fleet.send(id) — reachable from the master's send_worker tool (src/tools/fleet-tools.ts:91) and the desktop's worker.send (src/daemon/connection.ts:372) — passes requireLive on the idle (or lazy-resumable, via materialize) worker, and Worker.send()'s idle path immediately pushes the message and transitions to running (src/core/worker.ts:141-146). The SDK then executes Bash/Edit tools with cwd=worktree concurrently with the checkout, producing exactly the half-overwritten mixed state the guard's own comment says must be prevented. Worse, that send fires onTurnStart → checkpoint(id) which is serialized only on ckptChains (which restore never joins), so the new checkpoint runs `git add -A` on the mid-restore worktree and persists a corrupt restore point (addCheckpoint with a sha of an arbitrary mixture). Neither side ever learns the other ran: restore returns fleet.ack success, the turn keeps running.

**Trigger**: 1) A worker finishes a turn and goes idle. 2) Desktop user opens the CheckpointMenu and clicks restore to seq 0 → connection.ts:397 `await this.fleet.restore(id, 0)` passes the guard (status idle) and enters `git checkout <sha> -- .`. 3) Milliseconds later the home master (woken by a notify, or simply continuing its own turn) calls send_worker for the same worker → fleet.send → Worker.send at idle: queue.push + onTurnStart(checkpoint) + transition(running). 4) The SDK's first Edit/Bash writes files while git checkout is still rewriting the same paths → worktree is a mixture of checkpoint-0 files and new-turn files; the concurrently-taken checkpoint snapshots that mixture as the next 'restore point'. 5) Both the restore ack and the worker turn report success; the corruption is silent.

## 8. [MEDIUM] session.delete races in-flight/queued master turns and armed worker notifications — ghost SDK turn + FK violations on the deleted session

**Where**: `src/core/session-manager.ts:156` · verified 3/3 lenses

SessionManager.delete() awaits master.stop() (which resolves after abort()+interrupt(), NOT after the turn's drain/DB writes — MasterAgent.idle() exists for that but is never awaited) and removes the session from the live map only AFTER tearing down workers. Multiple races follow: (a) the aborted turn's drain records notice.interrupted via addSessionEvent after repos.deleteSession — FK violation, runTurn rejects with a SqliteError; (b) a turn QUEUED on turnChain is not cancelled at all and starts after deletion, throwing FK on addMessage and again in its catch; (c) worst, fleet.delete(w.id) → agent.stop() synchronously emits worker.status 'stopped', WorkerNotifier consumes the one-shot notify arm and — since the session is still in this.sessions — master.notifyWorker chains a flush doTurn that launches a REAL SDK query on a session mid-deletion (token burn); when the row cascades away, every write from that turn FK-throws, and the notifyWorker catch's addPendingNotification also FK-throws and is absorbed. Bus events (master.status running/idle) are emitted for a session clients just saw deleted. Fix direction: remove the session from the map / drain via master.idle() / suppress notifications before fleet cleanup and row deletion.

**Trigger**: 1) Master spawns a worker with spawn_worker notify:true (arm persisted at spawn). 2) While the worker is still running its first dispatch, the user deletes the session (session.delete). 3) delete() awaits master.stop(), then loops fleet.delete(w.id): Worker.stop() transitions to 'stopped' and synchronously emits worker.status. 4) WorkerNotifier consumes the arm; sessions.get(sessionId) still hits the live map → master.notifyWorker chains a flush turn; the idle master starts the flush doTurn on the next microtask while delete() awaits removeWorktree. 5) doTurn records the notice (row still present) and calls queryFn → real SDK turn begins. 6) delete() finishes: sessions.delete(id); repos.deleteSession(id) cascades. 7) The ghost turn's next DB write throws SQLITE FK; errors are swallowed; the SDK call was wasted. Variants: an in-flight aborted turn's drain or a queued user turn hits the same FK violations even with no workers.

## 9. [MEDIUM] Lazy resume (materialize) and fork silently drop maxTurns and effort — automation turn cap becomes unlimited after a daemon restart

**Where**: `src/core/fleet-orchestrator.ts:110` · verified 2/3 lenses

The worker loop's only runaway guard is the per-result maxTurns cap enforced in Worker.consume (worker.ts:361-369). maxTurns (and the spawn-time effort override) exist only as in-memory arguments to the original spawn: the workers table has no max_turns/effort columns, the orchestrator Entry struct has no such fields, so rehydrate() cannot restore them and materialize() calls the factory without them. In server.ts the subFactory then resolves `maxTurns: o.maxTurns` to undefined (= unlimited) and `effort: o.effort ?? settings.workerEffort()` (global default). Model and permission_mode ARE explicitly persisted 'for restart consistency', so this is an accidental gap. Consequence: an automation-spawned worker (whose max_turns is documented as the ONLY budget guard for unattended bypassPermissions runs) that survives a daemon restart as a lazy-resumable entry runs uncapped on its next send. fork() (fleet-orchestrator.ts:209-213) has the same gap: the forked lazy entry copies model/permissionMode but not maxTurns/effort.

**Trigger**: 1) Create an automation with a worker action and max_turns=10; it fires and fleet.spawn passes maxTurns:10 → Worker enforces the cap. 2) Restart the daemon while the worker is mid-task (or idle). 3) rehydrate() restores it as a lazy-resumable entry — Entry carries no maxTurns/effort. 4) send_worker → requireLive → materialize(id, e) calls deps.factory(...) with no maxTurns/effort. 5) server.ts builds the Worker with maxTurns undefined → the cap check `cap != null` never fires; the unattended worker runs unlimited turns with the global default effort, silently diverging from its automation config.

## 11. [MEDIUM] Armed notify (notify_armed) is never delivered after a daemon restart — master waits forever

**Where**: `src/core/fleet-orchestrator.ts:142` · verified 3/3 lenses

The notify arm is persisted in workers.notify_armed explicitly so it survives a restart (docs/architecture/fleet-lifecycle.md:62), and WorkerNotifier fires on failure settles specifically 'so the master isn't left waiting forever on a failed dispatch'. The daemon restarts (deploy/crash) while the worker is running or idle. 3. The worker sits settled with notify_armed=1 forever; the master (and the user, for headless automation sessions) never learns the dispatch outcome.

## 14. [MEDIUM] Daemon shutdown: fleet.close fires armed notifications into live masters — a ghost SDK turn races db.close and the notification is permanently lost

**Where**: `src/daemon/server.ts:330` · verified 3/3 lenses

startDaemon's close() runs `await fleet.close(5000); await sessions.drain(5000); db.close();`. fleet.close() stops every live worker, each of which synchronously emits worker.status 'stopped'. WorkerNotifier is still subscribed, so for any worker whose notify arm is set it atomically CONSUMES the arm and delivers. If the home master is live, notifyWorker chains a brand-new flush doTurn (a real SDK query) onto the turnChain — a full master turn launched DURING shutdown, with no closing guard in MasterAgent (unlike FleetOrchestrator's `closing` flag). sessions.drain waits at most 5s; a typical LLM turn takes longer, so db.close() fires mid-turn and every subsequent write throws 'database is not open'. **Trigger**: 1) Master spawns a worker with notify:true; the worker is still running. 2) User quits the desktop app / daemon receives SIGTERM → close() runs. 3) fleet.close calls agent.stop() on the running worker → transition('stopped') emits worker.status synchronously. 4) WorkerNotifier consumes the arm and delivers to the live master → notifyWorker chains a flush doTurn → real SDK query launched during shutdown. 5) sessions.drain(5000) times out before the LLM turn completes; db.close() executes. 6) The ghost turn's recordEvent throws 'database is not open'; notifyWorker's catch calls addPendingNotification on the closed DB, which throws and is swallowed. 7) On restart, pending_notifications is empty and the master never learns the dispatch settled — a master relying on notify:true waits forever.

## 15. [MEDIUM] Once-wakeup delete-before-fire: a daemon restart during (or right after) the fire permanently loses the wakeup with no trace

**Where**: `src/core/scheduler.ts:83` · verified 3/3 lenses

fireOnce deletes the `once` automation row from SQLite BEFORE dispatching it, and the dispatch then runs a full master turn that can take minutes.
