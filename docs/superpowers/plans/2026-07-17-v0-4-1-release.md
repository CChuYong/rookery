# v0.4.1 Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish rookery v0.4.1 from the current `origin/main` with accurate Slack-focused release notes and complete desktop artifacts.

**Architecture:** The annotated Git tag remains the release version source of truth, so source package versions stay unchanged and `.github/workflows/release.yml` stamps `0.4.1` during CI. The release metadata commit records the notes and procedure; the tag is created only after local gates and the matching `main` CI succeed. The tag-triggered workflow then builds and publishes signed/notarized macOS, Linux, and Windows artifacts.

**Tech Stack:** Git, GitHub CLI, GitHub Actions, Node.js 22, npm, Vitest, TypeScript, Electron Builder

## Global Constraints

- Release exactly the tested `origin/main` history after v0.4.0; do not rewrite or move the published v0.4.0 tag.
- Keep `package.json` source versions unchanged because the release workflow stamps versions from `v0.4.1`.
- Preserve the repository commit trailer: `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`.
- Do not tag until the release metadata commit has a successful `main` CI run.
- Do not declare completion until all platform assets and updater metadata are uploaded and stable latest-download URLs resolve to v0.4.1.

- [x] **Step 2: Push and wait for matching CI**

Run:

```bash

CI_RUN_ID="$(gh run list --workflow CI --commit "$RELEASE_SHA" --limit 1 --json databaseId --jq '.[0].databaseId')"
gh run watch "$CI_RUN_ID" --exit-status
```

Expected: the run whose `headSha` equals the release metadata commit completes with `success`.

- [x] **Step 3: Reconfirm the release target**

```bash

git rev-parse origin/main
```

Expected: clean `main`; both revisions are identical.
